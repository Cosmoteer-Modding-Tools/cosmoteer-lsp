# Schema seam

Context-aware behavior driven by **what a `.rules` value means in the game**, beyond the
structural reference/inheritance rules. The schema is extracted from `Cosmoteer.dll` (the C#
`*Rules` classes the game deserializes `.rules` into) and shipped here as data.

## Files

- **`cosmoteer.schema.json`**: the extracted schema bundle (~1.2 MB). Generated by the
  `schemagen` Mono.Cecil tool (see the project memory `schema-extraction-source`). Regenerate after
  a Cosmoteer update. Shape: `{ meta, registries, types, enums, unresolved }`, keyed by C# FullName.
- **`schema.types.ts`**: the TypeScript shape of the bundle (`SchemaBundle`, `ValueType`, …).
- **`cosmoteer-schema-json.d.ts`**: ambient module declaration that types the JSON import as
  `SchemaBundle` so the type checker does **not** infer a giant literal from the 1.2 MB file. The
  runtime is bundled by esbuild (which inlines the JSON). `tsc` only type-checks.
- **`schema.ts`**: the read side, `fieldsOf(class)` (walks `extends`), `fieldOf`, `enumDef`,
  `registryOf`, `classByDiscriminator(disc, registryHint?)`.
- **`schema-context.ts`**: the AST ⇄ schema bridge. `classOfGroup(group)` resolves a group's concrete
  class from its own `Type=` field. `registryForContainer(container)` infers which polymorphic
  registry a container's children belong to by reading a sibling's `Type` (so a part's `Components`
  and a bullet's `Components` disambiguate themselves).

## What consumes it

- **`features/completion/autocompletion.schema.ts`**: value completion. `Type = …` → the
  registry's discriminators; enum field → its members; boolean field → `true`/`false`.
  Registered in `autocompletion.service.ts`. Tests: `test/completion.schema.test.ts`.

## Model notes / gotchas

- A group's concrete class is selected by its **own `Type=` value** (the C# `[SerialBaseType]`
  dispatch), resolved against the registries. 15 discriminators collide across registries, so pass a
  `registryHint` (inferred from the container) to disambiguate.
- Containers like `Components` are **custom-deserialized** in the engine, so they have no
  `[Serialize]` field linking parent → registry; that link is recovered from siblings, not the schema.
- `getDocumentKind(uri)` (in `../document-kind.ts`) still only distinguishes `rules` vs `mod-rules`.
  Finer document-kind → root-class mapping (Part vs Shot vs Effect) is the next step for
  field-name (not just value) completion and whole-document schema validation.
