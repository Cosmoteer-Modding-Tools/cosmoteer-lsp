# Changelog

All notable changes to this project will be documented in this file.

# Version 0.4.0 Beta

-   Added **semantic-token highlighting** for `.rules` and made **`.shader` a first-class language**. The TextMate grammar stays the instant, offline base layer (it colours before the server connects and when no Cosmoteer path is set); semantic tokens are the overlay the editor paints on top once the file is parsed, replacing the grammar's regex guesswork with the real parse — a `&…` reference, a bareword enum value (`CombineMode = Add`), a math function (`ceil`), a `Key =` field name and a top-level entity declaration are each coloured by what they actually are. Because the colouring is computed on the server and delivered over LSP, it lights up identically in **both VS Code and JetBrains** (the native IntelliJ LSP API renders `textDocument/semanticTokens` since 2024.2.2) from one implementation, instead of a per-editor regex grammar. `.shader` files — previously opened as plain text — now get their own HLSL grammar, semantic tokens that colour the settable `_`-uniforms uniformly wherever they appear (plus types, keywords, preprocessor directives and functions), **hover** that explains whatever the cursor is on — a `_`-uniform's declared type (or that it is engine-provided), an HLSL intrinsic or texture method's signature and what it does (`lerp(a, b, s)`, `float4 Sample(sampler, uv)`), an HLSL type, or a function the file defines — **go-to-definition** that jumps from an `#include "…"` to the included file and from a `_`-uniform or function name to its declaration in the file or anywhere in its `#include` chain, a **document outline** of the file's uniforms and functions (breadcrumb + Outline view), **autocompletion** (the HLSL builtin types, keywords and intrinsic functions, plus the uniforms, functions and struct types the file **and its `#include` chain** declare, and the engine-provided uniforms such as `_texture`/`_time` that live in `base.shader`) that turns **context-aware after a `.`** — offering a vector's swizzles limited to its component count, a struct's members, or a texture's sampling methods (`Sample`/`SampleLevel`/`GetDimensions`), resolved from the base expression's type instead of the whole builtin list — each shown with its result type (`.xyz` on a `float4` is a `float3`, `Sample` returns a `float4`) and a one-line explanation — and **signature help** for HLSL intrinsic calls (`lerp(a, b, s)`, `clamp(x, min, max)`, …) **and for the shader's own functions** — a call to a function the file or its `#include` chain defines shows that function's real return type and typed parameters (`float4 loadRawNormals(float2 uv, float scale)`) with the active argument highlighted. Completion now also offers the **parameters and locals in scope** — inside a function body the enclosing function's parameters (`input`, `uv`) and the local variables declared before the cursor are suggested (with their type) alongside the builtins and uniforms, scoped to the enclosing function so another function's locals are not offered. And it **stops offering names where you are inventing one** — right after a type at the start of a declaration or parameter (`float x`, `(float2 uv`) the list is suppressed, since that identifier is a new variable name, not a reference (initializer positions on the right of `=` still complete normally). The shader features work in both editors too (the JetBrains plugin now sends `.shader` to the same server). In a Cosmoteer project (a workspace with `.rules` files) the extension automatically opens `.shader` files with the Cosmoteer Shader language even when VS Code's built-in Unity ShaderLab support also claims the `.shader` extension, so no manual language pick is needed (it only overrides the generic ShaderLab/plain-text default, honours the `cosmoteerLSPRules.associateShaderFiles` opt-out, and never runs the `.rules` validator on a shader)
-   Added **progress indicators while indexing**. The project-wide indexes the server builds on first use — the workspace symbol table, the cross-file reference id index, the inheritance-base index, and the alias-root index that walks the game data tree — now surface an LSP work-done progress notification (with a running file count) while they build, so a large project's first completion, find-references, or required-field check shows an "indexing" indicator instead of an unexplained pause. The build still runs to completion and caches even if the triggering request is cancelled, and it is a no-op for clients without progress support
-   Improved the **live WebGL shader preview**: non-square sprites are now letterboxed to their texture's aspect ratio instead of being stretched to the square canvas; the stage gained a **backdrop selector** (checkerboard / dark / light / grey, so an emissive or additive material can be judged against the right background), a **pause/play** toggle that freezes the animation clock, and a **blend-mode override** (normal / additive / premultiplied / multiply) for matching the game when the auto-resolved mode reads approximate. The preview also **updates live** as you edit — changing the material's constants/tint/blend or the shader code itself re-renders it automatically (debounced), and because the server reads open editor buffers it reflects **unsaved** `.shader` edits, not just what is on disk
-   Added full **schema intelligence / type safety** for `.rules` files. A schema of every `.rules` type — field names, value types, required/optional, defaults, enums and polymorphic `Type=` registries — is extracted directly from the game's own `*Rules` classes (via a Mono.Cecil tool in `tools/schemagen/`, with a reachability prune from the document root so non-`.rules` serialization is dropped) and shipped with the extension. It drives:
    -   **Field-name completion** offering the fields valid in the current group or whole-file root, each carrying its type / required / default / enum members as documentation and inserted as a ready-to-fill snippet (`Name = $0`, a `{ … }` block for groups, `[ … ]` for lists). Inherited base-class fields are included (abstract bases like `BaseQuadEffectRules` contribute `Sprite`/`Bucket`/… to every quad effect), and the class is resolved THROUGH inheritance — a `MyTurret : BaseTurret { … }` that doesn't redeclare its `Type` still completes the base's fields
    -   **Value completion** for `Type=` discriminators, enum fields and booleans — resolved in named groups, custom-deserialized containers (`Components`), typed list elements (`Layers [ { Type = … } ]`) and at a whole-file root's top level (a doodad/effect/music file's `Type`). Auto-pops after `=` (e.g. typing `Mode = ` immediately offers the enum members), even before any value is typed
    -   `Type` suggested first (with its discriminator set) in a polymorphic group that has not declared its subtype yet, and a one-pick "Insert N required fields" completion that scaffolds all still-missing required fields of the current class at once
    -   **Sibling component references** (`OperationalToggle = IsOperational`) completed from the other components in the same container, and **cross-file `ID<…>` references** (`ResourceType = battery`) completed from the project's whole-file-root declarations (cached, file-watcher-maintained index)
    -   **Validation** of invalid enum values, invalid boolean values, invalid `Type=` discriminators (in nested groups and at whole-file roots) — each with a "Did you mean …?" quick fix — a non-numeric value written into a numeric angle/scalar field (`Direction`/`Angle`/`Modifiable…`, e.g. `RotateSpeed = Fast`), and a value in an integer-only field that *resolves* (through math and references) to a fraction (e.g. `BlueprintArcSpriteSegments = (7 / 2)`); all verified false-positive-free across all 954 vanilla `Data/*.rules` files
    -   **Hover** showing a field's schema signature (type / required / default / enum members / reference target), the concrete class a `Type=` discriminator selects, and what a reference resolves to (the sibling component, or the declaring file for a cross-file id)
    -   **Go to Definition, Find All References and Rename** for both sibling and cross-file `ID<…>` references
    -   The **document outline** now annotates each group with the schema class it resolves to (e.g. `Turret → TurretWeaponRules`)
-   Added **missing-required-field validation** (`cosmoteerLSPRules.diagnostics.validateRequiredFields`, **on by default**): flags a polymorphic component group (`Type = TurretWeapon`, …) that omits a field the game requires, checking the full inheritance chain so a field supplied by a base never counts as missing. The hard part was knowing what is *actually* required: the schema extractor now derives a field's optionality from real C# signals — an explicit `Optional` attribute, a constructor-assigned default, a nullable annotation, an inline (empty-alias) member, or a collection type — instead of treating every un-attributed field as required, which had marked ~2300 fields required and produced false positives all over vanilla. Cross-file `BASE_*` templates are recognised and skipped via a project-wide, file-watcher-maintained index of inheritance-base names (so a base group completed by its deriving parts is never flagged), `~`-rooted runtime templates and any unresolvable base make the check stay silent, and the one genuinely engine-injected case left (a nebula doodad's spawner-supplied `ID`/`NebulaID`/`CategoryKey`) is allow-listed. Verified to **zero false positives across all 954 vanilla `Data` files and 42 real Steam Workshop mods (7820 files)**. Can be turned off to skip the one-time project index build it performs
-   Added **opt-in validation of cross-file GUI id references** (`cosmoteerLSPRules.diagnostics.validateCrossFileReferences`, default off). It flags a `ToggleID` / `ColorID` / `TargeterID` / `TriggerID` whose id is declared by no GUI collection in the project, with a "Did you mean …?" quick fix, so a typo such as `ToggleID = "on_of"` is caught. Scoped to those four GUI kinds on purpose, since their declarations are harvested completely from any file including a mod's own (verified zero warnings across all 954 vanilla files). Other reference kinds are left out because their declarations are distributed in ways a central index cannot fully see (per-part stats, runtime types, or buffs whose mod additions are only reached through the game's own aliases), which would cause false positives
-   Added **part-category completion**. A part category has no declaration file, it exists because parts name it (`Category = armor`, `TypeCategories = [armor, …]`), so completion now offers every category used across the project. List element positions such as `TypeCategories = [ … ]` complete too, not only direct `Field = ` values
-   Extended cross-file `ID<…>` references to **map keys** — in a `map<reference X, V>` collection the key itself is the reference (`MaxBuffValues = { Engine = 100% }` keys on a buff, `StatusResistances { cosmoteer.disruptor_drain = … }` keys on a status). Go-to-definition, find-all-references and rename on the key all work, and completing a new key offers the project's X ids. Renaming an entity now also rewrites its map-key uses, not only its value-position uses. Only the map's keys are treated as references, never the collection field's own name
-   Extended cross-file `ID<…>` intelligence to **group-name-keyed entities** — entities declared as the NAMED members of a `map<reference X, V>` collection rather than as list elements: buffs (`buffs.rules` → `Engine { … }`, referenced by `ReceivableBuffs = [ Engine ]`), part features, … The authoritative collections are identified via the alias index (only a collection the game root actually aliases counts, never a per-object `map<reference X, scalar>` consumer field), so completion and go-to-definition resolve these too
-   Added **alias-based fragment rooting**: `cosmoteer.rules` pulls most game data in from fragment files by aliasing a field to them (`Buffs = &<buffs.rules>`, `Factions = &<factions.rules>/Factions`, `CareerMode = &<career.rules>`), so a fragment file has no self-describing root class. The extension now walks those aliases forward from the game root (recursing through whole-file group aliases) and roots each fragment by the field that pulls it in — making schema completion, hover, validation, go-to-definition and references work *inside* fragment files (a tech's `Prerequisites = [ … ]` referencing another tech in the same `techs.rules`, a buff group's fields in `buffs.rules`, …). This also disambiguates same-named lists by their source (a `Techs` reached via `career.rules` is a `TechRules`, not a pvp `BuildBattleTechRules`). Verified false-positive-free: schema validation stays at zero warnings across all 954 vanilla files with fragments rooted
-   Extended cross-file `ID<…>` references to **list-element positions** — a reference written inside a list, e.g. `AllowedFactions = [fringe]`, `Features = [ … ]`, `ReceivableBuffs = [ Engine ]`, `Prerequisites = [ tech ]` — so completion, go-to-definition, find-references and rename now work for those, not just direct `Field = ref` values. This also fixed a foundational bug where the cursor on any element inside a `Field = [ … ]` / `Field = { … }` resolved to the whole container instead of the inner element (so hover/completion/navigation now land on list elements everywhere)
-   Extended cross-file `ID<…>` intelligence to **aggregate list-element entities** — entities defined as elements of a list reachable from the game root (`Cosmoteer.Data.Rules`) rather than as whole-file roots: factions (`factions.rules`), GUI part toggles/colors/targeters/triggers and their choices, career-mode techs/encounters/difficulties/sector-types, ship doors, background styles, and more. Which list fields hold such entities, and how each element is identified (its `ID`, or a self-`…ID` field like `ToggleID`/`ColorID`), is derived from the schema itself — so completion (`RemainderFaction = ` → every faction id), go-to-definition (jumps into `factions.rules`/`part_toggles.rules`), find-all-references and rename now all work for these references, including ones nested inside parts and ones whose target is nested (a toggle's `Choices`). Group-keyed kinds (damage types, part categories) and `Key`/`Value` lists (ship render layers) are a later phase
-   Added **opt-in component-reference validation** (`cosmoteerLSPRules.diagnostics.validateComponentReferences`, default off): flags a component `ID<…>` value (e.g. `OperationalToggle = IsOperational`) whose id names no component anywhere in the part or its inherited bases, with a "Did you mean …?" quick fix. Existence is the part-wide union of every component id declared in the file plus all inherited base parts (matching how the engine looks ids up part-wide, so a reference deep in a nested sub-component still resolves a top-level or inherited component). Off by default because a single-file check cannot be fully false-positive-free: some components are injected by the engine at runtime (never declared in `.rules`, e.g. `ConstructionTracker`), and some `ID<PartComponent>` fields are used as opaque keys rather than real references (e.g. a shared `OverridePriorityKey`)
-   Added **signature help** for math/function calls in values (`Damage = ceil(&Base / 2)`): typing inside a function's parentheses shows its parameter list and highlights the argument being typed. Curated signatures with descriptions for the evaluatable functions (`pow(base, exponent)`, `round(x, places)`, the trig/log family, variadic `min`/`max`/`sum`/`avg`, Cosmoteer's `db2vol`), and a generic signature for any other valid mXparser function. Works mid-edit and on nested calls (`floor(sqrt(…))` highlights the inner `sqrt`) via a raw-text scan that reconstructs the call nesting independently of the parser
-   Added classification of whole-file-root documents (a shot file is a `BulletRules`, a resource file a `ResourceRules`, an effect/music/doodad file is dispatched by its top-level `Type`), each self-validated against the document's own fields to avoid mis-rooting
-   Extended the schema to the **Halfling engine types** so particle and effect files get completion, hover and validation *inside* their bodies, not only at the top level. The extractor now also reads `HalflingCore.dll`, so a particle file's whole `Def { … }` — its `Initializers`/`Updaters` (`Type = SetRandom`, `Operator`, `ColorRamp`, …), `Renderer` (`StandardQuadRenderer`, …) and `Material` (`Texture`, `TargetBlendMode`, …) — now resolves, where previously everything below the file's top-level fields was unknown. This covers the engine registries whose `Type=` dispatch lives on an interface (particle updaters/renderers) and stays false-positive-free across all 954 vanilla files. The extractor also now models any class that merely carries `[Serialize]` members (not only those tagged `[ReflectiveSerialization]` at the class level) as a group, maps `Direction` to its true numeric (angle) type, and injects a few curated group types for custom-deserialised structs that carry no serialization attributes (e.g. a part's `VirtualInternalCells` pairs, and the engine's `AtlasSprite` — the `{ File … Size … VertexColor … AnimationInterval … }` group every sprite field is written as, ~5700 vanilla uses), registers registry members that dispatch by plain subclass name with no attribute (the ship-generator stages: `Type = AsteroidStage`/`GridStage`/…), supplies a field-level overlay for the alternate spellings a custom deserialiser also accepts (`IntColor`'s float `Rf`/`Gf`/`Bf`/`Af`), resolves the abstract sprite/material interface slots (`ISprite`/`IMaterial`/`IAnimatedSprite`) to their concrete classes so the full field set (a `Sprite`'s `Texture`/`Shader`/…, an animation's `FrameCount`/`FrameSize`/…) completes, and recognises the inline **shader constants** a material accepts — the `_`-prefixed uniform keys (`_hotColor = …`) written alongside its `Shader`, which hover labels as shader constants — resolving a further batch of previously-opaque fields so they too get completion, hover and validation. Further rounds of curation closed the largest remaining buckets — the empty-alias inline members (a blueprint sprite's `AtlasSprite`, every proxy part's `ProxyRules` → `ComponentID`/`PartLocation`/…), a mode toggle's `OnMode`/`OnModes`, a toggled-components part's `Components`, the resource converter's `From*`/`To*` storages and quantities, the thruster fuel/force fields, the buff/status/ship/editor-group references, the doodad-spawner and particle-emitter members, and `PartRules`' `EditorGroup`/`Flammable` — all read by custom deserializers schemagen cannot reflect, supplied via the additive overlay (an `OVERLAY_INLINE_TYPES` mechanism copies an embedded empty-alias type's fields into its container). Most of those custom-read fields are now recovered automatically: `schemagen` parses the **IL of each type's deserializer methods** for the generic `reader.TryReadFromPath<T>("Name")` / `ReadFromPath<T>` / `ReadOptionalFromPath<T>` calls, taking the OT key from the literal string and the value type from the generic argument — so a class read by a custom constructor contributes its fields without any hand-curation, and they stay correct across game updates. A registry-resolution fix was also made so a polymorphic group with an ambiguous discriminator (a shot's `GlowSprite { Type = Sprite }` vs a part's `Sprite { Type = Sprite }`) resolves to the right class for its context (`BulletSpriteRules` vs `PartSpriteRules`), from the other components in its container. The extractor also gained **C# code-mod support**: passing a mod's assemblies via `--mod <dll-or-dir>` merges their serializable types into the registries, so a mod's custom components/effects (`Type = AmmoChange`, …) get completion and validation too — making the same tool usable both by the game developers on a new release and by players running code mods (see `tools/schemagen/README.md`). Two further resolution fixes raised coverage again: a field is now recognised under any of its `[Serialize(AlternateAliases=…)]` spellings (vanilla writes e.g. `LeftEdgeEffect` for the C# member `LeftAdd`), and the career/pvp/creative **spawner-generator files** now root correctly — the `&<>`-included `SimulationGenerator`/`GalaxyGenerator` fragments and the sector files — so their sub-spawners resolve within the spawner registry (`Type = Nebula` → `NebulaSpawner`) instead of the colliding doodad one; this also taught discriminator resolution to follow a base registry to the derived registry that actually declares the `Type` (a `List<SimSpawner>` slot whose elements are `SimObjectSpawner`s). A whole-vanilla coverage test tracks this: the schema now recognises **98.6%** of every field written across the 954 `Data` files (most of the residual is dead copy-paste fields in vanilla's own particle updaters that the engine ignores), and the same measured across 42 real Steam Workshop mods rose from 94.3% to **98.8%**, all still false-positive-free (verified again here: zero new warnings across 7820 mod files)
-   Added **shader intelligence and a live WebGL shader preview**. A material's `Shader = "x.shader"` defines a set of `_`-prefixed uniform constants the material can set (`_hotColor`, `_z`, `_litReflectiveStrength`, …); their names live in the `.shader` file, not the schema. The extension now parses the referenced `.shader` (following `#include`s and dropping the engine-bound built-ins such as `_texture`/`_time`/`_transform` and the global-lighting uniforms), so a material group **completes** its shader's actual constants (typed, with the declared default) and **hovers** an inline `_`-key with its HLSL type and default. On top of that, a **Preview shader** CodeLens above every `Shader = …` (and a `Cosmoteer: Preview Shader` command) opens a side panel that renders the material with WebGL the way the game does: it resolves the sprite's texture, translates the shader's HLSL pixel stage to GLSL ES (preprocessing `#include`/`#ifdef`, converting `Texture2D.Sample`/`cbuffer`/intrinsics, coercing literals, and pruning code unreachable from the pixel entry point), applies the material's blend mode and colour tint, and exposes every constant as a live control (colour pickers for colours, sliders for numbers) that re-renders as you drag. The translator strips comments and HLSL attributes, dead-code-eliminates everything unreachable from the pixel entry point, and handles `float4 pix`/geometry/beam/post-process pixel stages, so **140 of 149 vanilla shaders translate cleanly** (the remaining 9 are include-only `base_*` scaffolds with no pixel entry point that a material never references directly); anything it still cannot handle falls back to a faithful textured-and-tinted render rather than a broken one. The translator also lowers `.SampleLevel`/`.GetDimensions` and `Texture2D` parameters, rewrites HLSL integer casts (`(int2)(…)`), renames GLSL-reserved locals (`half`), and emits the `GL_OES_standard_derivatives` extension directive (enabled in the preview runtime) so the decal and distortion shaders link. Also added **opt-in shader-constant validation** (`cosmoteerLSPRules.diagnostics.validateShaderConstants`, default off): flags a `_`-constant the referenced shader declares no uniform for (a typo such as `_hotColr`, with a did-you-mean quick fix) and one whose value is the wrong shape for its type (a scalar uniform given a list, a colour given a boolean). The type check is false-positive-free on vanilla; it is opt-in only because the game itself ships a few dead constant keys its shaders do not read. Also added **opt-in in-shader diagnostics** (`cosmoteerLSPRules.diagnostics.validateShaderCode`, default off) that check the `.shader` file itself: an `#include` whose target does not exist, a `_`-prefixed uniform read that no file in the include chain declares (a probable typo such as `_gloww`), and a call to a function that is neither an HLSL intrinsic, a builtin constructor, a `#define`d macro, nor a function the shader or its includes define. It is a lexical check, not an HLSL compiler, and stays conservative: the undeclared-uniform/function checks run **only when the whole `#include` chain is readable** (a missing base include means the symbol set is partial, so it reports just the unresolvable include and skips the rest rather than risk a false positive), `_`-prefixed locals and parameters and engine-bound uniforms are recognised as declared, and a root-anchored `./Data/…` include is left unjudged when no game path is configured. The same opt-in check also validates **calls and declarations against the shader's own function signatures**: a call to a file/include function with the wrong number of arguments (`blend(x)` where `blend` takes two — skipped for an overloaded name whose arity is ambiguous), a declaration that assigns a call's result to an incompatible type (`float x = loadRawNormals(2, 2)` where the function returns `float4`, an HLSL truncation — a scalar splatting into a vector is allowed and only a lone call as the whole initializer is judged), and a type used as an assignment target with no variable name (`float = …`)
-   Added **dual-form support for `Modifiable<T>` fields** (`ModifiableFloat`/`Int`/`Time`/`Angle`, 250+ fields). A value like `Damage` can be written either as a plain scalar (`Damage = 5`) *or* as a group carrying the unmodified `BaseValue` plus inline buff/status/effect-scale modifiers and `MinValue`/`MaxValue` clamps (`Damage { BaseValue = 5  BuffType = …  BuffMode = Multiply }`). The scalar form keeps its numeric completion/inlay/validation, and the group form now resolves to a curated `ModifiableValue` class — so opening the `{ }` offers `BaseValue`/`Modifiers`/`BuffType`/`BuffMode`/… with hover and enum validation (e.g. an invalid `BuffMode` is flagged). Verified false-positive-free across all 954 vanilla files. This generalises a new schema mechanism (a scalar value type carrying a `groupForm`) also used for image `Texture` fields written as a `{ File … }` group
-   Added **particle data channel** intelligence. A particle updater reads and writes named data channels — `SetRandom { DataOut = rot_vel }` declares the channel `rot_vel` and `Operator { BIn = rot_vel }` uses it (the engine's `ParticleDataID`, detected by schema type, with direction from the `…Out`/`…In`/`…InOut` field-name suffix). Completion offers the channel names the file already uses, and go-to-definition (jumps to the writer), find-all-references and rename all work for a channel across the whole file (a channel written in `EmitterDef` and read in `Def` resolves across both)
-   Added **colour field hints and swatches**. A float colour group serialises its components as `Rf`/`Gf`/`Bf`/`Af` (not the C# `R`/`G`/`B`/`A`), so completion and hover inside a `Color { … }` now offer the right names; and every `{ Rf Gf Bf Af }` (0–1) or `{ R G B A }` (0–255) colour group renders an inline **colour swatch** whose picker rewrites the component values in place
-   Modelled more **inline engine value types**. A particle updater's range/value (`SetRandom`, `Lerp`, `SetValue`) is custom-deserialised with no reflected fields, but its `ValueType` + `FromValue`/`ToValue`/`Value` are written directly in the updater group — these are now schema fields, so `ValueType` gets enum completion/validation and the rest get hints (the polymorphic component values stay permissive to avoid false positives). **Hotkeys** (`Hotkey`/`DefaultHotkey = [Control, N]`) are now a list of key names with completion and typo validation. Both verified false-positive-free across all 954 vanilla files
-   Go-to-definition, find-all-references and rename now work on a `&`-reference **embedded inside a math expression** (`ToQuantity = (&HEAT_PER_INTERVAL) * (&~/HEAT_TO_RESOURCE)`, `Damage = ceil(&Base / 2)`), not only when the reference is the whole value. The cursor→node finder behind go-to-definition descends into expression and function-call operands (matching the hover/reference finder), so a reference used in computed-stat math resolves, gets its hover, and is rewritten consistently on rename. Verified the reverse search already enumerates these so rename never leaves a dangling reference
-   Improved **asset-path completion**, which previously only fired on *quoted* values that already carried a file extension. Two fixes: (1) an already-classified asset (sprite/sound/shader, by extension) now completes whether or not it is quoted — the engine accepts unquoted asset paths and vanilla/mods write them that way (`Shader { File = particle_normals.shader }`, `Sprite = part.png`), including inside a `mod.rules` `Overrides` block; (2) completion is now **schema-aware** — when the field is known to be an asset (a `Shader`/`Sound`/sprite `File`, including the dual-form `File` inside a `Shader { … }` / `Texture { … }` group, whose kind is taken from the group's own asset slot), the same-folder assets are offered *while the path is still being typed*, before any extension is present, and filtered to that asset's kind. The extension-less look-alike heuristic still requires quotes, so completion never fires on ordinary name fields
-   Fixed three robustness issues that surfaced on particle/effect files: an autocomplete crash on an empty `Key = ` value, a document-outline error (`selectionRange must be contained in fullRange`) from bare keys and empty values, and the colour picker overwriting a group's brace instead of its component values
-   Improved Rename safety: a rename searches the whole game tree so cross-file references resolve, but the resulting edits are now restricted to the open mod — the read-only vanilla `Data` install is never written to (opt out via `cosmoteerLSPRules.rename.allowEditingVanillaFiles` if you are a developer working directly on the game data)
-   Fixed a Windows path bug where files read from disk (backslash paths) were not recognized by the path-based document-root rules, so cross-file features missed disk-only files
-   Added full `mod.rules` manifest support: parsing, validation (unknown verbs, missing required fields, target existence) and autocompletion (verb names, fields per verb and target paths) for `Actions`
-   Added Go to Definition (cross-file, follows references and inheritance)
-   Added Find All References across the mod and the game `Data` tree
-   Added Rename / Refactoring of a symbol across files
-   Added Document Symbols (Outline / breadcrumbs)
-   Added Workspace Symbols (search symbols across the project)
-   Added Hover information showing the resolved value of a reference, with sprite image preview for assets
-   Added Inlay Hints showing computed math/function results inline (e.g. `Damage = (&Base)/2 + ceil(17/2)  = 14`), using mXparser semantics
-   Added autocompletion for assets (image/sound/shader paths)
-   Added "Did you mean …?" quick fixes for mistyped references and asset paths
-   Added optional whole-workspace validation: validate every `.rules` file in the open folder(s), not just the open file (opt-in via the `cosmoteerLSPRules.diagnostics.validateWholeWorkspace` setting, off by default to keep memory usage low). The bulk pass discards each file's AST after producing diagnostics (rather than caching it) so it stays memory-bounded on large projects
-   Added multi-root workspace support: navigation, find-all-references, rename and workspace validation now span all open workspace folders
-   Added evaluation of percentage literals (`50%` → `0.5`) in hover and inlay hints

-   Fixed several parser crashes and edge cases: nested function-call arguments, scientific-notation numbers (`3.4E+38`), time literals (`30:00`), top-level inline math expressions (`1/16`) and EOF handling
-   Fixed a false `"7-" is not a known math function` error: a `-` or `/` before a space-separated parenthesized group (`7- (12/64)`) is now lexed as a subtraction/division operator instead of being glued into a bogus function name
-   Fixed a large class of false `Reference name is not known` warnings found by revalidating real mods:
    -   File-URI decoding now handles any drive letter and case (previously only a lowercase `C:` drive decoded; an uppercase or non-`C:` drive made the directory walk fail and flagged every relative reference)
    -   Group-merge globals (`SW_PARTICLES = &<a.rules>, &<b.rules>`) now resolve members from ALL merged files, not just the first
    -   Runtime-root (`&~/…`) references are no longer statically validated (they resolve at instantiation against a context the static file does not contain)
    -   A reference whose final member exists but whose value is a runtime `~` reference (e.g. `&../DamagePerShot`) is no longer flagged
    -   Case-insensitive `&<./data/…>` references into the merged game `Data` tree now resolve against the game install
    -   References into a vanilla file the mod patches with a whole-file or file-aliasing-global `Overrides` action now resolve their mod-added members: `&/INDICATORS/<modMember>` (target `<…/indicators.rules>`) and `&/COMMON_EFFECTS/<modMember>` (target `<cosmoteer.rules>/COMMON_EFFECTS`, a global aliasing a file) are reached through the vanilla global the reference travels by — in diagnostics, Go to Definition (which now jumps to the mod's override source, not the vanilla file) and autocompletion (which now lists the mod-added members alongside the file's own)
    -   Backslash-separated file reference paths (`&<hit_effects\foo.rules>`) now resolve: the game's path grammar accepts `\` and .NET resolves it on Windows, so these are valid references. The lexer keeps `\` inside a `<…>` path (it is whitespace elsewhere) instead of splitting the reference, and the resolver normalizes it to `/` for the directory walk
    -   Fixed a startup race where a `.rules` file already open when the extension activated could build the mod's effective-game-tree context before the game-install scan finished — caching an empty result that made every override reference (`&/INDICATORS/…`, …) false-flag until a manifest changed. The context is now dropped once the scan completes
