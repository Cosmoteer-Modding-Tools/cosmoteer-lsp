# Changelog

All notable changes to this project will be documented in this file.

# Version 0.4.0 Beta

-   Added **code formatting** for `.rules` files (including `mod.rules` and its actions, which are the same ObjectText syntax) and `.shader` files (Format Document in VS Code / JetBrains). ObjectText makes classic pretty-printing dangerous — unquoted values may contain spaces (`Name = Big Gun`), a newline ends a value unless a `\` suppressed it, and `10 - 3` lexes differently from `10-3` — so the formatter never moves, merges or splits tokens. It re-lexes the document with the real lexer, treats every token, comment and `\` continuation as untouchable, and rewrites only the whitespace around them: indentation from `{`/`[` nesting (tab or spaces per editor settings, continuation lines one level deeper, comments indented with their surroundings), exactly one space around `=` and the inheritance `:`, none before and one after `,`/`;`, none inside `(…)`/`[…]`, one inside an inline `{ … }`, trailing whitespace trimmed, blank-line runs capped at two and exactly one final newline. Multi-line strings, `@"…"` verbatim strings and block-comment interiors are preserved byte-for-byte, tab-aligned unquoted values keep their tabs (a space would merge them into one value), and CRLF files stay CRLF. As a hard guarantee the result is lexed again and must produce the identical token stream and comments, otherwise the formatter changes nothing — verified across all 954 vanilla files and 7820 workshop-mod files (zero changes rejected on well-formed files, zero idempotence divergence; the single holdout is a mod file that is one unterminated `/*` comment, which is exactly what the guard is for). The `.shader` formatter re-indents by brace depth (wrapped argument lists one extra level, preprocessor directives at column 0) and never touches anything inside a line. Formatting is also safe mid-edit: incomplete syntax (an unclosed brace or call, a dangling `A =`, an unterminated string) formats conservatively or not at all, verified by a character-by-character typing simulation. Two settings control it: `cosmoteerLSPRules.formatting.enabled` (default on) switches the feature, and `cosmoteerLSPRules.formatting.formatOnSave` (default off) auto-formats right before every save via LSP `willSaveWaitUntil`, independent of the editor's own `editor.formatOnSave`
-   All math-function knowledge now lives in **one registry** (`math-function-registry.ts`): every function's name, argument-count bounds, parameter names, documentation and (where we model it) numeric implementation in a single data table that signature help, the unknown-function diagnostic, arity checking, argument-type checking and the resolved-value evaluator all derive from. When the game's math vocabulary changes (a Cosmoteer update registering a new `IMathFunction`, or an mXparser upgrade), the update is a one-line data edit in that file — no feature code needs touching
-   **Arity checking now covers every known math function**, not only the evaluatable subset: the mXparser vocabulary was already grouped by argument count, so the registry carries bounds for all of them and a forgotten argument (`root(&A)`, `if(cond, then)`) is flagged even for functions we cannot evaluate. Still too-few-only by design, since the parser's call flattening can inflate an apparent argument count but never deflate it. Signature help improves the same way: a valid-but-unmodelled function now shows its real parameter count (`root(a, b)`) instead of a generic `(…)`
-   **Commented-out references no longer count as reachable.** The file-reachability closure (mod overview + `workspaceValidationScope: modRulesReachable`) scanned raw text, so a parts list whose registrations were commented out still marked the whole disabled folder reachable, and its files kept getting validated. Comments are now stripped before refs are followed, mirroring the game lexer's rules (`//`, `/* */`, and no comment starts inside `"…"` or `@"…"` strings, so a `"http://…"` value cannot swallow the rest of its line). On the Star Wars mod this honestly retires 1156 additional dead files (verified by hand across every cut-point family: all were genuinely disabled registrations). The overview's ← annotation now also names the reachable file whose commented-out line disables a chain, which is exactly the line to revive it
-   Fixed **circular-inheritance diagnostics landing on the wrong lines**. The cycle detection walks inheritance chains across files, and when the reference that closes a loop lives in another file (a derived part inheriting a base that inherits itself), the error was anchored to that foreign node — its offsets were then mapped through the validated document's text, producing diagnostics at arbitrary, shifting line numbers. A cycle found inside a foreign chain is now reported on the validated file's own inheritance reference that leads into it, so the squiggle always sits on the line that actually pulls the cycle in
-   Members of a **map-typed slot** now resolve through the map's value type, so their `Type =` dispatches in the map's declared registry instead of falling back to sibling inference. The regression case was a `ToggledComponents` sub-components map whose first child writes `Type = ArcShield`: the discriminator collides between the part-component and media-effect registries, and the wrong one won, so a fragment file included as `Components = &<shield_1_blue.rules>/Components` got media-effect completions (and a bogus "Insert 3 required fields") inside its components. The container's slot is resolved once and shared between its class resolution and the map fallback, since recursing separately for each doubles the work per nesting level and blows up on deep files.
-   Shader constants written in **group form** now complete their fields: a `Texture2D` uniform written as `_waveTex { … }` offers the texture fields (`File`, `UVMode`, `SampleMode`, `MipLevels`, …) and a `float4` colour uniform written as a group offers the colour fields (`Rf`/`Gf`/`Bf`/`Af`). The group is not a schema field, so its class is resolved from the constant's kind in the material's referenced `.shader` file. This never worked, not even on vanilla files.
-   A known token in a position no grammar rule accepts (the stray `=` in `X = &<a>, = &<b>`) is now reported as `Unexpected "="` instead of the misleading "Unknown token type / this could be a parser bug". Verified against the game: `HalflingCore.dll`'s `OTGroupNode.Parse` throws the same `Unexpected "=" at position …` for this input, so it is invalid ObjectText, not a parser gap. (Group-level multi-value assignments like `X = &<a>, &<b>` are ALSO rejected by the game's parser — a comma there is a node terminator and the next node must start with an identifier — but we deliberately stay lenient on those because they appear in modder-convenience files the game never loads, e.g. a mod-root `cosmoteer.rules` whose aliases are actually injected via `mod.rules` actions.)
-   A mod's root `cosmoteer.rules` is no longer force-seeded into the **reachability closure**. The game applies manifest actions to its own `Data/cosmoteer.rules` and never opens the mod's copy (verified in `Cosmoteer.Data.Assets` / `ModInfo.ApplyPreLoadMods`), so with `workspaceValidationScope = modRulesReachable` that file is now honestly skipped, and it only counts as reachable when reached content actually references it. The mod overview lists it under unreachable files with a note explaining the convention instead of implying it was forgotten. Editor navigation still overlays its globals (mod-context), which is independent of the closure.
-   The cross-file and shader validators are now **on by default**: component references (`validateComponentReferences`), cross-file GUI ids (`validateCrossFileReferences`), shader constants (`validateShaderConstants`), in-shader diagnostics (`validateShaderCode`) and localization keys (`validateLocalizationKeys`). Each pass earned its default by scanning to zero false positives across the vanilla install, and across 42 workshop mods every remaining finding was hand-triaged as a genuine mod bug (missing strings keys, `BeerMug1` referencing a component declared as `BeerMug01`, dangling ids in dead template files, shader includes the game itself could not resolve). The cross-file passes hold off until the game install is indexed, since without it a reference to something vanilla declares would be flagged wrongly. What that took: the component-id union now understands assignment-form (`PowerToggle={…}`) and reference-valued (`BeerMug10=&~/Part/BeerMug`) component declarations, folds in include-valued components blocks (vanilla's missile launcher declares its variant components via `Components = &<he/…>/Components`), skips files other files inherit from (their references resolve in the deriving part), skips `ChainableProxy` groups (their `ComponentID` lives in the chained part) alongside the runtime-injected `ConstructionTracker` and the two non-sibling fields `OverridePriorityKey` / `ChainFireToggleComponent`; GUI toggle/color/targeter/trigger ids declared as loose groups a `mod.rules` action adds to the game list are now indexed (declaration recognized by shape: `Style`/`Choices`/button members and no `Type`); the strings index now harvests list elements by index (`FameTitles/0`); shader includes unresolved next to a mod file now retry at the same mod-relative path inside the game tree (mods mirror the `Data` layout); the shader arity check honors HLSL default parameters and ignores preprocessor directives and `[attribute]` contexts; and the handful of dead constant keys vanilla itself ships are skipped. Whole-workspace validation stays opt-in (it is a memory/CPU trade, not a correctness one)

-   Added a **Mod overview** report and a **reachability-scoped validation mode**. A "Show mod overview" CodeLens on `mod.rules` (also `Cosmoteer: Show Mod Overview` in the palette) renders what the manifest actually does as a markdown preview: the header fields, every action with its verb, name, target, source and flags, and whether each target resolves in the effective game tree — an action whose target resolves to nothing silently does nothing in game, so those are marked ✗ and counted. Below that, a **file-reachability section** computes which `.rules` files the game can actually load (the closure of the manifest's action sources, their `<…>` includes and inheritance, plus the strings folder) and lists the unreachable rest grouped by folder — backups and templates are expected there, but a part or effect the modder meant to ship is a bug caught before release (on the 4652-file Star Wars mod the report surfaced a finished reactor part no parts list registers). Each unreachable file that is referenced only from other unreachable files carries a `← referencer` annotation, separating dead chains (one commented-out include kills a whole folder of fragments, and rewiring the chain's root revives every file behind it) from files nothing references at all. The same closure powers the new `cosmoteerLSPRules.diagnostics.workspaceValidationScope` setting (`allFiles` default, `modRulesReachable` opt-in), which restricts the whole-workspace validation pass to the files the game loads so dead content stops flooding the Problems panel; files open in the editor always validate
-   The shader preview now **runs a shader's own vertex stage**. When a `.shader` defines its own `vert` (85+ vanilla shaders and most Star Wars: A Cosmos Divided effect shaders do), the preview no longer replaces it with generic stand-in values: the translator also converts the vertex stage to GLSL, synthesizes its input struct from the preview quad (sprite locations over a nominal world span, particle center/offset/rotation/scale, beam start/direction/length/thickness with the intensity and fade sliders feeding the real inputs), runs the actual `vert`, and forwards every output field to the pixel stage as varyings. Per-vertex math the game computes — a beam's thickness-over-intensity, scrolling uv setups, screen-space locations, `_transform`-projected positions — now reaches `pix` the way the engine feeds it, and the world-to-clip transform is fitted per input family so the result still fills the canvas. Translation gained `sincos` lowering (the beam helpers call it) for this. A vertex stage that cannot be synthesized (unknown input fields, geometry-stage shaders) or fails to compile falls back to the previous fixed-quad render, and the status line shows a "vertex stage" tag when the real one is running. Verified structurally across all 112 SW-mod and 149 vanilla shaders: every generated stage defines all functions it calls and writes every varying its fragment reads
-   Fixed the shader preview failing to compile a shader that calls `base.shader`'s `multiplyAdditiveLightValue`, whose signature carries an HLSL default parameter value (`float nrmlStrengthLimit = 1.0`). GLSL forbids parameter initializers, so the translator now strips them from the definition and emits a forwarding overload per omitted trailing argument, matching HLSL call resolution. Vanilla never calls the function (dead-code elimination removed it, hiding the gap), but workshop lighting shaders such as Star Wars: A Cosmos Divided's `thruster_plume_light.shader` do, and they previously fell back to the approximate textured render
-   Added **semantic-token highlighting** for `.rules` and made **`.shader` a first-class language**. The TextMate grammar stays the instant, offline base layer. It colours before the server connects and when no Cosmoteer path is set. Semantic tokens are the overlay the editor paints on top once the file is parsed, replacing the grammar's regex guesswork with the real parse. A `&…` reference, a bareword enum value (`CombineMode = Add`), a math function (`ceil`), a `Key =` field name and a top-level entity declaration are each coloured by what they actually are. Because the colouring is computed on the server and delivered over LSP, it lights up identically in **both VS Code and JetBrains** from one implementation, instead of a per-editor regex grammar (the native IntelliJ LSP API renders `textDocument/semanticTokens` since 2024.2.2). `.shader` files, previously opened as plain text, now get their own HLSL grammar. Their semantic tokens colour the settable `_`-uniforms uniformly wherever they appear, plus types, keywords, preprocessor directives and functions. **Hover** explains whatever the cursor is on: a `_`-uniform's declared type (or that it is engine-provided), an HLSL intrinsic or texture method's signature and what it does (`lerp(a, b, s)`, `float4 Sample(sampler, uv)`), an HLSL type, or a function the file defines. **Go-to-definition** jumps from an `#include "…"` to the included file, and from a `_`-uniform or function name to its declaration in the file or anywhere in its `#include` chain. A **document outline** lists the file's uniforms and functions (breadcrumb + Outline view). **Autocompletion** offers the HLSL builtin types, keywords and intrinsic functions, plus the uniforms, functions and struct types the file **and its `#include` chain** declare, and the engine-provided uniforms such as `_texture`/`_time` that live in `base.shader`. It turns **context-aware after a `.`**, offering a vector's swizzles limited to its component count, a struct's members, or a texture's sampling methods (`Sample`/`SampleLevel`/`GetDimensions`), resolved from the base expression's type instead of the whole builtin list. Each is shown with its result type (`.xyz` on a `float4` is a `float3`, `Sample` returns a `float4`) and a one-line explanation. **Signature help** covers HLSL intrinsic calls (`lerp(a, b, s)`, `clamp(x, min, max)`, …) **and the shader's own functions**: a call to a function the file or its `#include` chain defines shows that function's real return type and typed parameters (`float4 loadRawNormals(float2 uv, float scale)`) with the active argument highlighted. Completion also offers the **parameters and locals in scope**. Inside a function body the enclosing function's parameters (`input`, `uv`) and the local variables declared before the cursor are suggested (with their type) alongside the builtins and uniforms, scoped to the enclosing function so another function's locals are not offered. And it **stops offering names where you are inventing one**: right after a type at the start of a declaration or parameter (`float x`, `(float2 uv`) the list is suppressed, since that identifier is a new variable name, not a reference (initializer positions on the right of `=` still complete normally). The shader features work in both editors too (the JetBrains plugin now sends `.shader` to the same server). In a Cosmoteer project (a workspace with `.rules` files) the extension automatically opens `.shader` files with the Cosmoteer Shader language, even when VS Code's built-in Unity ShaderLab support also claims the `.shader` extension, so no manual language pick is needed. It only overrides the generic ShaderLab/plain-text default, honours the `cosmoteerLSPRules.associateShaderFiles` opt-out, and never runs the `.rules` validator on a shader
-   Added **progress indicators while indexing**. The server builds several project-wide indexes on first use: the workspace symbol table, the cross-file reference id index, the inheritance-base index, and the alias-root index that walks the game data tree. These now surface an LSP work-done progress notification (with a running file count) while they build, so a large project's first completion, find-references, or required-field check shows an "indexing" indicator instead of an unexplained pause. The build still runs to completion and caches even if the triggering request is cancelled, and it is a no-op for clients without progress support
-   Added **localization-key intelligence** for `KeyString` fields (`NameKey`, `DescriptionKey`, …). A key such as `NameKey = base.turret/name` now **completes** from the strings the project declares (its own `/strings/` plus the game's), **hovers** to show the resolved text in every available language, and an **opt-in validator** (`cosmoteerLSPRules.diagnostics.validateLocalizationKeys`, default off) flags a key that no strings file declares (case-insensitively) with an **insert-into-all-strings quick fix** that adds the missing key to every language file at once. The string index is file-watcher-maintained and verified false-positive-free across vanilla and 42 Steam Workshop mods
-   Added **deprecation hints**. A registry of renamed types (seeded with the `Ammo*` → `Resource*` rename) is wired into `Type=` discriminator validation and hover, so a deprecated type name is flagged with its replacement and a one-click quick fix that swaps it in
-   Added **reverse-include rooting** so unrooted fragment files (a particle `_def.rules`, an effect fragment, …) gain full schema intelligence on their own. The extension indexes which field `&`-includes each fragment and roots the fragment from that field, so completion, hover, validation, go-to-definition and the shader preview all work inside a fragment opened standalone. Always on, with a schema-layer fallback hook, and verified false-positive-free on vanilla (16 files rooted, 0 warnings)
-   Added rooting for the **`builtin_ships/**` files** as `BuiltinShipsDatabase` (via overlay field-adds and a document-root path rule), so a built-in ship definition gets schema completion, hover and validation. Zero false positives across vanilla and mods
-   Improved **startup performance**: the project-wide indexes are now built in a single directory walk and warmed up in the background, the alias-root fixpoint resolves alias-only, and file reads are done read-ahead, so a large project reaches its first completion and validation faster (also fixed a pre-existing client hang from uncached workspace folders)
-   Added full **schema intelligence / type safety** for `.rules` files. A schema of every `.rules` type — field names, value types, required/optional, defaults, enums and polymorphic `Type=` registries — is extracted directly from the game's own `*Rules` classes (via a Mono.Cecil tool in `tools/schemagen/`, with a reachability prune from the document root so non-`.rules` serialization is dropped) and shipped with the extension. It drives:
    -   **Field-name completion** offering the fields valid in the current group or whole-file root, each carrying its type / required / default / enum members as documentation and inserted as a ready-to-fill snippet (`Name = $0`, a `{ … }` block for groups, `[ … ]` for lists). Inherited base-class fields are included (abstract bases like `BaseQuadEffectRules` contribute `Sprite`/`Bucket`/… to every quad effect), and the class is resolved through inheritance — a `MyTurret : BaseTurret { … }` that doesn't redeclare its `Type` still completes the base's fields
    -   **Value completion** for `Type=` discriminators, enum fields and booleans — resolved in named groups, custom-deserialized containers (`Components`), typed list elements (`Layers [ { Type = … } ]`) and at a whole-file root's top level (a doodad/effect/music file's `Type`). Auto-pops after `=` (e.g. typing `Mode = ` immediately offers the enum members), even before any value is typed
    -   `Type` suggested first (with its discriminator set) in a polymorphic group that has not declared its subtype yet, and a one-pick "Insert N required fields" completion that scaffolds all still-missing required fields of the current class at once
    -   **Sibling component references** (`OperationalToggle = IsOperational`) completed from the other components in the same container, and **cross-file `ID<…>` references** (`ResourceType = battery`) completed from the project's whole-file-root declarations (cached, file-watcher-maintained index)
    -   **Validation** of invalid enum values, invalid boolean values, invalid `Type=` discriminators (in nested groups and at whole-file roots) — each with a "Did you mean …?" quick fix — a non-numeric value written into a numeric angle/scalar field (`Direction`/`Angle`/`Modifiable…`, e.g. `RotateSpeed = Fast`), and a value in an integer-only field that *resolves* (through math and references) to a fraction (e.g. `BlueprintArcSpriteSegments = (7 / 2)`); all verified false-positive-free across all 954 vanilla `Data/*.rules` files
    -   **Hover** showing a field's schema signature (type / required / default / enum members / reference target), the concrete class a `Type=` discriminator selects, and what a reference resolves to (the sibling component, or the declaring file for a cross-file id)
    -   **Go to Definition, Find All References and Rename** for both sibling and cross-file `ID<…>` references
    -   The **document outline** now annotates each group with the schema class it resolves to (e.g. `Turret → TurretWeaponRules`)
-   Added **missing-required-field validation** (`cosmoteerLSPRules.diagnostics.validateRequiredFields`, **on by default**): flags a polymorphic component group (`Type = TurretWeapon`, …) that omits a field the game requires, checking the full inheritance chain so a field supplied by a base never counts as missing. The hard part was knowing what is *actually* required: the schema extractor now derives a field's optionality from real C# signals — an explicit `Optional` attribute, a constructor-assigned default, a nullable annotation, an inline (empty-alias) member, or a collection type — instead of treating every un-attributed field as required, which had marked ~2300 fields required and produced false positives all over vanilla. Cross-file `BASE_*` templates are recognised and skipped via a project-wide, file-watcher-maintained index of inheritance-base names (so a base group completed by its deriving parts is never flagged), `~`-rooted runtime templates and any unresolvable base make the check stay silent, and the one genuinely engine-injected case left (a nebula doodad's spawner-supplied `ID`/`NebulaID`/`CategoryKey`) is allow-listed. Verified to **zero false positives across all 954 vanilla `Data` files and 42 real Steam Workshop mods (7820 files)**. Can be turned off to skip the one-time project index build it performs
-   Added **opt-in validation of cross-file GUI id references** (`cosmoteerLSPRules.diagnostics.validateCrossFileReferences`, default off). It flags a `ToggleID` / `ColorID` / `TargeterID` / `TriggerID` whose id is declared by no GUI collection in the project, with a "Did you mean …?" quick fix, so a typo such as `ToggleID = "on_of"` is caught. Scoped to those four GUI kinds on purpose, since their declarations are harvested completely from any file including a mod's own (verified zero warnings across all 954 vanilla files). Other reference kinds are left out because their declarations are distributed in ways a central index cannot fully see (per-part stats, runtime types, or buffs whose mod additions are only reached through the game's own aliases), which would cause false positives
-   Added **part-category completion**. A part category has no declaration file, it exists because parts name it (`Category = armor`, `TypeCategories = [armor, …]`), so completion now offers every category used across the project. List element positions such as `TypeCategories = [ … ]` complete too, not only direct `Field = ` values
-   Extended cross-file `ID<…>` references to **map keys** — in a `map<reference X, V>` collection the key itself is the reference (`MaxBuffValues = { Engine = 100% }` keys on a buff, `StatusResistances { cosmoteer.disruptor_drain = … }` keys on a status). Go-to-definition, find-all-references and rename on the key all work, and completing a new key offers the project's X ids. Renaming an entity now also rewrites its map-key uses, not only its value-position uses. Only the map's keys are treated as references, never the collection field's own name
-   Extended cross-file `ID<…>` intelligence to **group-name-keyed entities** — entities declared as the named members of a `map<reference X, V>` collection rather than as list elements: buffs (`buffs.rules` → `Engine { … }`, referenced by `ReceivableBuffs = [ Engine ]`), part features, … The authoritative collections are identified via the alias index (only a collection the game root actually aliases counts, never a per-object `map<reference X, scalar>` consumer field), so completion and go-to-definition resolve these too
-   Added **alias-based fragment rooting**: `cosmoteer.rules` pulls most game data in from fragment files by aliasing a field to them (`Buffs = &<buffs.rules>`, `Factions = &<factions.rules>/Factions`, `CareerMode = &<career.rules>`), so a fragment file has no self-describing root class. The extension now walks those aliases forward from the game root (recursing through whole-file group aliases) and roots each fragment by the field that pulls it in — making schema completion, hover, validation, go-to-definition and references work *inside* fragment files (a tech's `Prerequisites = [ … ]` referencing another tech in the same `techs.rules`, a buff group's fields in `buffs.rules`, …). This also disambiguates same-named lists by their source (a `Techs` reached via `career.rules` is a `TechRules`, not a pvp `BuildBattleTechRules`). Verified false-positive-free: schema validation stays at zero warnings across all 954 vanilla files with fragments rooted
-   Extended cross-file `ID<…>` references to **list-element positions** — a reference written inside a list, e.g. `AllowedFactions = [fringe]`, `Features = [ … ]`, `ReceivableBuffs = [ Engine ]`, `Prerequisites = [ tech ]` — so completion, go-to-definition, find-references and rename now work for those, not just direct `Field = ref` values. This also fixed a foundational bug where the cursor on any element inside a `Field = [ … ]` / `Field = { … }` resolved to the whole container instead of the inner element (so hover/completion/navigation now land on list elements everywhere)
-   Extended cross-file `ID<…>` intelligence to **aggregate list-element entities** — entities defined as elements of a list reachable from the game root (`Cosmoteer.Data.Rules`) rather than as whole-file roots: factions (`factions.rules`), GUI part toggles/colors/targeters/triggers and their choices, career-mode techs/encounters/difficulties/sector-types, ship doors, background styles, and more. Which list fields hold such entities, and how each element is identified (its `ID`, or a self-`…ID` field like `ToggleID`/`ColorID`), is derived from the schema itself — so completion (`RemainderFaction = ` → every faction id), go-to-definition (jumps into `factions.rules`/`part_toggles.rules`), find-all-references and rename now all work for these references, including ones nested inside parts and ones whose target is nested (a toggle's `Choices`). Group-keyed kinds (damage types, part categories) and `Key`/`Value` lists (ship render layers) are a later phase
-   Added **opt-in component-reference validation** (`cosmoteerLSPRules.diagnostics.validateComponentReferences`, default off): flags a component `ID<…>` value (e.g. `OperationalToggle = IsOperational`) whose id names no component anywhere in the part or its inherited bases, with a "Did you mean …?" quick fix. Existence is the part-wide union of every component id declared in the file plus all inherited base parts (matching how the engine looks ids up part-wide, so a reference deep in a nested sub-component still resolves a top-level or inherited component). Off by default because a single-file check cannot be fully false-positive-free: some components are injected by the engine at runtime (never declared in `.rules`, e.g. `ConstructionTracker`), and some `ID<PartComponent>` fields are used as opaque keys rather than real references (e.g. a shared `OverridePriorityKey`)
-   Added **signature help** for math/function calls in values (`Damage = ceil(&Base / 2)`): typing inside a function's parentheses shows its parameter list and highlights the argument being typed. Curated signatures with descriptions for the evaluatable functions (`pow(base, exponent)`, `round(x, places)`, the trig/log family, variadic `min`/`max`/`sum`/`avg`, Cosmoteer's `db2vol`), and a generic signature for any other valid mXparser function. Works mid-edit and on nested calls (`floor(sqrt(…))` highlights the inner `sqrt`) via a raw-text scan that reconstructs the call nesting independently of the parser
-   Added classification of whole-file-root documents (a shot file is a `BulletRules`, a resource file a `ResourceRules`, an effect/music/doodad file is dispatched by its top-level `Type`), each self-validated against the document's own fields to avoid mis-rooting
-   Extended the schema to the **Halfling engine types** so particle and effect files get completion, hover and validation *inside* their bodies, not only at the top level. The extractor now also reads `HalflingCore.dll`, so a particle file's whole `Def { … }` — its `Initializers`/`Updaters` (`Type = SetRandom`, `Operator`, `ColorRamp`, …), `Renderer` (`StandardQuadRenderer`, …) and `Material` (`Texture`, `TargetBlendMode`, …) — now resolves, where previously everything below the file's top-level fields was unknown. This covers the engine registries whose `Type=` dispatch lives on an interface (particle updaters/renderers) and stays false-positive-free across all 954 vanilla files. The extractor also now models any class that merely carries `[Serialize]` members (not only those tagged `[ReflectiveSerialization]` at the class level) as a group, maps `Direction` to its true numeric (angle) type, and injects a few curated group types for custom-deserialised structs that carry no serialization attributes (e.g. a part's `VirtualInternalCells` pairs, and the engine's `AtlasSprite` — the `{ File … Size … VertexColor … AnimationInterval … }` group every sprite field is written as, ~5700 vanilla uses), registers registry members that dispatch by plain subclass name with no attribute (the ship-generator stages: `Type = AsteroidStage`/`GridStage`/…), supplies a field-level overlay for the alternate spellings a custom deserialiser also accepts (`IntColor`'s float `Rf`/`Gf`/`Bf`/`Af`), resolves the abstract sprite/material interface slots (`ISprite`/`IMaterial`/`IAnimatedSprite`) to their concrete classes so the full field set (a `Sprite`'s `Texture`/`Shader`/…, an animation's `FrameCount`/`FrameSize`/…) completes, and recognises the inline **shader constants** a material accepts — the `_`-prefixed uniform keys (`_hotColor = …`) written alongside its `Shader`, which hover labels as shader constants — resolving a further batch of previously-opaque fields so they too get completion, hover and validation.
